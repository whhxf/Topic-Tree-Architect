# 产品需求说明书 (PRD)：AI 话题树构建与落地系统

> **产品代号**：Topic Tree Architect
> **版本**：v2.0
> **理论基础**：HubSpot 话题集群模型 + Koray Tuğberk GÜBÜR 语义权威地图（详见 `营销方法论.md`）

---

## 1. 产品概述

### 1.1 产品愿景

打造一个基于大语言模型的**对话式内容策略智能体**。通过"启发式教练对话 + 真实世界搜索数据"，引导用户挖掘产品上下文，生成具备 SEO 排名潜力和集客转化率的结构化"话题树（Topic Tree）"，并提供优先级排序、内容简报、内链规划的全流程落地指导。

### 1.2 目标用户

| 用户画像 | 核心需求 | 使用频率 |
|---|---|---|
| 独立开发者 / 早期创业者 | 懂产品但不懂内容营销，需要结构化指导 | 低频（1-2 棵树/季度） |
| B2B SaaS 市场营销人员 | 快速建立行业话题权威度 | 中频（每月迭代） |
| SEO 专家 / 内容团队 | 自动化关键词调研，替代传统手工流程 | 高频（多客户并行） |

### 1.3 核心价值主张

- **摒弃盲目填表**：以"资深商业顾问"的对话形式，动态挖掘产品真实痛点与差异化
- **结合真实数据**：通过 API 调用实时搜索引擎/SEO 数据，验证搜索意图与竞争度，杜绝纯 AI 幻觉
- **系统化输出**：输出带优先级评分的结构化（树干-树枝-树叶）策略，而非零散文章标题
- **覆盖全漏斗**：内容自动映射到 TOFU/MOFU/BOFU 转化阶段，确保流量与转化并重

### 1.4 本产品的竞品分析

| 竞品 | 优势 | 劣势 | 我们的差异化 |
|---|---|---|---|
| MarketMuse | 成熟的内容评分体系 | 价格极高（$600+/月），无对话引导 | 对话式体验 + 低门槛定价 |
| Clearscope | SERP 分析精准 | 仅优化单篇文章，无树状结构 | 系统级话题树，非单篇优化 |
| Frase | AI 内容简报能力强 | 缺乏战略层面的话题规划 | 先战略后执行的完整工作流 |
| WriterZen | 关键词聚类功能 | 无产品上下文挖掘，纯工具 | 业务顾问式交互 + 数据验证 |
| ChatGPT/Claude 裸用 | 免费灵活 | 无搜索数据支撑，输出质量不稳定 | 内置 SEO 数据 + 结构化输出保障 |

## 2. 双市场运营规范：国内市场 vs 海外国际市场

产品服务两类用户群体——面向国内市场的用户，和面向海外市场的用户。两者在搜索引擎生态、数据源、内容载体、SEO 逻辑、转化路径上存在**根本性差异**，直接影响系统的数据源接入、Tool 参数设计和 Prompt 策略。

用户在 S1 阶段回答"目标市场"时，系统即完成市场判定，并在 S2-S5 阶段全程切换对应的数据源和策略建议。

---

### 2.1 核心差异对照表

| 维度 | 国内市场（CN） | 海外市场（Global） |
|---|---|---|
| **主要搜索引擎** | 百度（~60%）、必应 CN、360搜索、神马 | Google（~92%）、Bing |
| **搜索数据工具** | 百度指数、5118、爱站网、百度搜索资源平台 | Keywords Everywhere、Ahrefs、SEMrush、Google Search Console |
| **话题树主要载体** | 独立站/官网博客 + 微信公众号矩阵 + 知乎专栏 | 独立站/官网博客（核心）|
| **内容分发渠道** | 微信公众号、知乎、小红书、B站、抖音、头条号/百家号 | LinkedIn、Twitter/X、YouTube、Reddit、Medium |
| **内链的 SEO 权重** | 中等（百度对内链依赖弱于 Google，外链仍有价值） | 极高（Google Pillar-Cluster 策略的核心基础） |
| **外链的 SEO 权重** | 较高（百度仍重视友链和外链质量） | 中等（Google 已大幅降低低质外链影响） |
| **内容权威度信号** | 原创标签、更新频率、收录速度、作者认证 | E-E-A-T（经验/专业/权威/可信）、语义实体覆盖 |
| **关键词研究逻辑** | 以百度指数衡量热度，5118 挖掘长尾词 | 以月搜索量（MSV）+ 关键词难度（KD）综合决策 |
| **用户搜索行为差异** | 部分用户习惯在平台内搜索（小红书/知乎/抖音），搜索引擎流量分散 | Google 是绝对主入口，用户习惯在 Google 完成发现→评估→决策 |
| **转化路径** | 公众号私信 → 企微加好友 → 私域运营 → 成交 | 邮件订阅（Lead Magnet）→ 邮件序列培育 → Demo 预约 → 成交 |
| **内容格式偏好** | 图文（公众号）、问答长文（知乎）、短视频（抖音/B站）、笔记种草（小红书） | 长篇博客文章（2000-4000 字）、YouTube 视频、Podcast、LinkedIn 长文 |
| **效果追踪工具** | 百度搜索资源平台、百度统计、微信公众号数据后台 | Google Search Console、Google Analytics 4 |
| **账号与支付** | 手机号/微信登录，微信支付/支付宝 | Email/Google 登录，信用卡/Stripe |

### 2.2 数据源差异：S2 阶段的分叉点

S2 阶段调用 `analyze_market_search_intent` 时，必须根据市场类型切换完全不同的数据源。

**国内市场数据源方案**：

| 数据需求 | 工具/API | 获取数据 | 局限性 |
|---|---|---|---|
| 关键词热度趋势 | 百度指数 API（需百度合作资质）或抓取 | 相对搜索热度趋势图 | 无绝对搜索量，只有相对指数 |
| 长尾关键词挖掘 | 5118 API 或 爱站 API | 长尾词列表、相关词 | 数据更新频率不如 SEMrush |
| 竞品内容分析 | 百度搜索结果抓取（搜狗 MIP 等） | 排名前 10 页面标题/描述 | 需处理反爬 |
| 社交平台热门话题 | 微信指数（公众平台接口）、知乎热榜 API | 社媒话题热度 | 各平台数据孤岛，需分开调用 |
| **MVP 降级方案** | **Tavily Search（目标站点设为 baidu.com）** | 百度搜索结果文本 | **无搜索量数据，需手动补充** |

**海外市场数据源方案**（延续 Section 3 既有设计）：

| 数据需求 | 工具/API | 获取数据 |
|---|---|---|
| 搜索量 + KD | Keywords Everywhere API | 月搜索量、竞争难度、CPC |
| 长尾词发现 | Tavily Search API | 真实搜索结果 + 相关问题 |
| SERP 竞品分析 | Tavily Search API | 排名前 10 页面特征 |

> **MVP 阶段实现策略**：优先完整支持海外市场的数据链路；国内市场在 MVP 阶段以 Tavily（指向百度）+ LLM 知识库作为降级方案，在 Phase 2 接入 5118 API 或百度指数，并在节点上标注"搜索量数据参考价值有限，建议结合百度指数人工核验"。

### 2.3 话题树结构差异：内容载体决定树的形态

国内市场的话题树需要考虑**多平台分发**，树的结构与海外有本质不同。

**海外市场：以独立站为中心的单树结构**

```
官网博客（唯一权威根基）
├── Pillar Page（树干）：长篇终极指南（3000+ 字）
│   └── Cluster / Leaf（树枝/树叶）：长篇深度文章（1500-2500 字）
└── 内链网络将所有内容绑定 → Google 视为完整知识体系
```

**国内市场：以官网为核心、多平台并行的森林结构**

```
官网博客（SEO 权重根基）
├── Pillar Page（树干）：百度 SEO 优化长文（2000+ 字，含目录和关键词布局）
│   └── Cluster / Leaf：细分长尾文章（800-1500 字，百度更倾向原创/更新频率）
│
知乎专栏（品牌权威背书，搜索引擎亦收录）
├── 同一树干话题的"问答版"——与官网文章差异化，不直接复制
│   └── 知乎回答中植入官网链接（外链 + 导流）
│
微信公众号（私域流量主阵地）
├── 公众号文章是官网文章的"精华提炼版"，通常 800-1200 字
│   └── 文末引导关注/加企微，进入私域转化漏斗
│
小红书/B站/抖音（TOFU 种草内容）
└── 将树叶内容切片为短视频/笔记，引流至官网或公众号
```

**系统实现要求**：
- 国内市场模式下，`generate_topic_tree_json` 需为每片树叶额外输出 `platform_versions` 字段，说明该内容如何适配到知乎/公众号/小红书等不同平台
- 树叶的 `estimated_word_count` 按平台分别给出（官网版/知乎版/公众号版）

**需在 JSON Schema 中新增的字段（仅国内市场激活）**：

```json
{
  "leaf_title": "...",
  "platform_versions": {
    "official_site": {
      "word_count": 1500,
      "focus": "百度SEO，关键词自然密度，含目录"
    },
    "zhihu": {
      "word_count": 800,
      "format": "问答体",
      "focus": "解答具体问题，建立专业形象，末尾植入官网链接"
    },
    "wechat_oa": {
      "word_count": 1000,
      "format": "图文，含封面图",
      "focus": "精华提炼，情感共鸣，末尾引导加企微"
    },
    "xiaohongshu": {
      "word_count": 300,
      "format": "笔记+配图（9张）",
      "focus": "种草，痛点共鸣，引导评论互动"
    }
  }
}
```

### 2.4 SEO 逻辑差异：内链 vs 外链的权重不同

这是两个市场在技术 SEO 上最重要的差异，直接影响 S4（内链规划）阶段的产品逻辑。

**Google（海外）的排名逻辑**：

- 内链是传递 PageRank 的核心机制，Pillar-Cluster 模型依赖内链才能成立
- 语义覆盖（Semantic Coverage）极为关键——文章中的实体词和 LSI 关键词密度直接影响排名
- E-E-A-T（经验 Experience / 专业 Expertise / 权威 Authoritativeness / 可信 Trustworthiness）是隐性排名因子，需要作者署名、外部引用、产品独家数据来建立
- 外链质量>数量，低质量外链可能触发惩罚

**百度（国内）的排名逻辑**：

- 原创度和更新频率是重要信号，百度有原创保护机制，"首发"优势明显
- 外链仍有效，尤其是来自权威域名（知乎/36kr/虎嗅等）的外链价值极高
- 内链有价值但依赖度低于 Google，百度对"知识图谱"的构建逻辑与 Google 有差异
- 收录速度是运营重点——百度主动推送（IndexNow/Sitemap）比 Google 更敏感
- 用户行为数据（停留时长、跳出率）对百度排名的影响比 Google 更显著

**对产品功能的影响**：

| 功能模块 | 海外市场行为 | 国内市场行为 |
|---|---|---|
| S4 内链规划 | 严格按 Pillar→Cluster→Leaf 的三层内链矩阵执行，这是 SEO 成败的关键 | 内链矩阵同样生成，但同时给出外链获取建议（推荐投稿平台：知乎/36kr/少数派） |
| 内容简报（Tool 3） | 强调 E-E-A-T 信号：建议加入原创数据/客户案例/作者 Bio | 强调原创度：建议先于竞品发布，加入百度收录推送建议 |
| 优先级评分 | 以 Google KD + 月搜索量为核心权重 | 以百度指数热度 + 竞品 PC 端排名难度为核心权重 |
| 效果追踪 | 集成 Google Search Console | 集成百度搜索资源平台 + 百度统计 |

### 2.5 转化路径差异：私域 vs 邮件列表

两个市场的"流量如何变成用户/客户"路径完全不同，直接影响每片树叶的 CTA 策略。

**海外市场的标准转化漏斗**：

```
Google 搜索 → 落地到博客文章（树叶）
    → 文中插入 CTA：免费下载 Checklist / 白皮书（Lead Magnet）
    → 用户留 Email → 进入邮件自动化序列（Drip Campaign）
    → 邮件 #3-5 发送 Demo 邀请 → 注册/付费
```

**国内市场的标准转化漏斗**：

```
百度/知乎/小红书搜索 → 落地到内容页
    → 文末/文中插入 CTA：扫码关注公众号 / 加企微领取资料
    → 进入企微私域 → 1v1 运营 or 社群培育
    → 提供 Demo / 试用邀请 → 注册/付费
```

**对 `cta_strategy` 字段的影响**：

```json
// 海外市场树叶的 CTA 策略
{
  "cta_type": "lead-magnet | free-trial | demo-request | newsletter",
  "cta_copy": "Download the free VAT compliance checklist →",
  "cta_placement": "文末 + 文中自然锚点"
}

// 国内市场树叶的 CTA 策略
{
  "cta_type": "wechat-follow | enterprise-wechat | resource-download | trial",
  "cta_copy": "扫码加企微，免费领取《XX行业内容营销实战手册》",
  "cta_placement": "文末 + 文章三分之二处 + 公众号文章底部",
  "wechat_ecosystem": {
    "official_account": true,
    "enterprise_wechat": true,
    "mini_program": false
  }
}
```

### 2.6 S1 阶段问题的差异化策略

当系统在 S1 阶段识别到用户目标市场后，需切换不同的追问角度。

**判定逻辑**：用户回答"目标市场是中国大陆"或提及"百度/公众号/知乎"等平台 → 切换为国内市场模式。

| S1 问题维度 | 海外市场追问角度 | 国内市场追问角度 |
|---|---|---|
| 内容发布渠道 | "您主要通过独立博客还是也会用 LinkedIn/Medium？" | "您目前内容主要发在官网博客、公众号，还是知乎/小红书？哪个渠道转化最好？" |
| 目标关键词难度 | "您的主要竞品在 Google 排名第几位？" | "在百度搜索您的核心词，竞品的百度权重大概是多少？" |
| 现有内容资产 | "请问您的独立站目前有多少篇博客文章？" | "您的官网博客、公众号历史文章各大概有多少篇？是否有知乎专栏？" |
| 转化路径 | "您目前用什么方式获取 Leads？是 Email 表单还是直接注册？" | "您目前是通过公众号引流、企微加好友，还是直接落地页转化？" |
| 竞品分析 | "您的竞品在 Google 上主要靠哪类内容获得流量？" | "您的竞品在百度/知乎/公众号上主要靠哪类内容获得曝光？" |

## 3. 核心业务流程（状态机）

系统维护一个 **状态机（State Machine）**，包含五个状态。每个状态有明确的准入条件、系统行为和退出条件。

```
S1 启发挖掘 → S2 数据校验 → S3 结构生成 → S4 内链规划 → S5 迭代与落地
     ↑                                                          |
     └──────────────── 用户要求重新开始 ─────────────────────────┘
```

### S1: 启发挖掘（Elicitation）

- **准入条件**：用户发起新的话题树构建请求
- **系统行为**：扮演资深商业顾问，通过递进式追问收集产品上下文。每轮只问 1-2 个问题。
- **信息采集维度**（按顺序）：
  1. 产品一句话介绍 + 产品形态（SaaS / 实体 / 服务）
  2. ICP（理想客户画像）：谁在用？什么行业？什么职级？
  3. 核心痛点：用户不用你的产品，最难受的是什么？
  4. USP（差异化卖点）：同类产品那么多，用户为什么选你？
  5. 竞品/替代方案：用户目前在用什么替代方案？
  6. 目标市场与语言：面向哪些国家/地区？内容用什么语言发布？
  7. 已有内容资产：是否已有博客/网站？有多少已发布文章？
- **退出条件**：以上维度收集完毕，且用户确认上下文总结无误
- **输出物**：结构化的 `ProductContext` 对象

### S2: 数据校验（Validation & Research）

- **准入条件**：S1 输出的 `ProductContext` 已获用户确认
- **系统行为**：
  1. 从 `ProductContext` 提取核心实体词（core_entities）
  2. 调用 `analyze_market_search_intent` 获取真实搜索数据
  3. 向用户展示数据摘要，包括：发现的高价值长尾词 Top 20、竞品已覆盖话题、内容缺口机会
  4. 询问用户是否需要调整方向或补充搜索维度
- **退出条件**：用户确认数据方向正确
- **输出物**：带有搜索量和竞争度标注的 `ValidatedTopicPool`

### S3: 结构生成（Generation）

- **准入条件**：S2 输出的 `ValidatedTopicPool` 已获用户确认
- **系统行为**：
  1. 结合 ProductContext + ValidatedTopicPool
  2. 使用 Pillar-Cluster 模型编排话题树
  3. 为每个节点标注搜索意图类型、漏斗阶段、优先级评分
  4. 调用 `generate_topic_tree_json` 输出结构化数据
  5. 前端渲染思维导图
- **退出条件**：话题树 JSON 成功渲染到前端画布
- **输出物**：完整的 `TopicTree` JSON

### S4: 内链规划（Link Architecture）

- **准入条件**：S3 输出的话题树已生成
- **系统行为**：
  1. 自动生成树内节点间的内链矩阵（树叶→树干、树干→树叶、树叶↔树叶）
  2. 若用户提供了已有网站 Sitemap，调用 `import_existing_sitemap` 将现有文章映射到话题树
  3. 标记"缺失链接"和"孤岛页面"
- **退出条件**：内链矩阵生成完毕
- **输出物**：`LinkMatrix` + 孤岛页面报告

### S5: 迭代与落地（Execution）

- **准入条件**：S3/S4 完成
- **系统行为**：
  1. 用户可在画布上交互式"剪枝"或"扩写"任意分支
  2. 用户点击某片"树叶"可触发 `generate_content_brief` 生成内容简报
  3. 用户可通过对话要求调整优先级排序
  4. 支持导出话题树为 JSON / CSV / Markdown 格式
- **退出条件**：用户主动结束会话或导出成果
- **输出物**：最终版话题树 + 内容简报 + 内链方案

## 4. 数据源策略

数据源是话题树质量的天花板。系统采用**分层数据架构**，按 MVP → 成熟期逐步升级。

### 3.1 MVP 阶段数据源

| 数据需求 | 选用方案 | 理由 | 月成本预估 |
|---|---|---|---|
| 长尾搜索词发现 | **Tavily Search API** | 针对 AI Agent 优化的搜索 API，返回结构化结果 | ~$50（1000次/月） |
| 搜索量 + 关键词难度 | **Keywords Everywhere API** | 性价比最高的搜索量数据源，支持批量查询 | ~$10（100,000 credits） |
| SERP 竞品内容分析 | **Tavily Search API**（复用） | 获取排名前 10 的页面标题和摘要 | 已含上方 |

### 3.2 成熟期数据源升级路径

| 数据需求 | 升级方案 | 触发条件 |
|---|---|---|
| 精准搜索量 + 趋势 | SEMrush API 或 Ahrefs API | 付费用户 > 100 或 B2B 客户需求 |
| 社交媒体热门讨论 | Reddit API + Twitter/X API | 用户反馈需要社交维度数据 |
| 已有网站分析 | Google Search Console API（用户授权） | Phase 3 反馈循环功能上线 |

### 3.3 数据返回标准格式

所有数据源的返回结果必须统一转换为以下内部格式：

**`priority_score` 计算逻辑**（1-10 分）：
- 搜索量权重 30%：月搜索量越高越好
- 竞争度权重 25%：KD 越低越好
- 商业意图权重 25%：BOFU > MOFU > TOFU
- 话题关联度权重 20%：与产品 USP 的语义关联度

**`search_intent` 四分类定义**：

| 意图类型 | 定义 | 典型修饰词 | 漏斗阶段 |
|---|---|---|---|
| informational | 用户想学习/了解 | 什么是、如何、为什么、指南 | TOFU |
| commercial | 用户在比较/评估 | 最好的、A vs B、评测、推荐 | MOFU |
| transactional | 用户准备购买/行动 | 购买、价格、注册、下载、免费试用 | BOFU |
| navigational | 用户在寻找特定品牌/产品 | [品牌名]、[产品名] + 登录/官网 | BOFU |

## 6. 数据模型与持久化

### 5.1 核心数据实体

```
User (用户)
├── id, email, name, plan_type, created_at
│
├── Project (项目，一个用户可有多个项目)
│   ├── id, user_id, project_name, product_context_json, created_at, updated_at
│   │
│   ├── TopicTree (话题树，一个项目可迭代多个版本)
│   │   ├── id, project_id, version, tree_json, link_matrix_json, status, created_at
│   │   └── ContentBrief (内容简报，挂在树叶节点下)
│   │       └── id, tree_id, leaf_id, brief_json, created_at
│   │
│   └── ConversationHistory (对话历史)
│       └── id, project_id, messages_json, state, created_at, updated_at
│
└── ExistingSite (已有网站)
    └── id, user_id, sitemap_url, pages_json, last_crawled_at
```

### 5.2 存储方案

| 数据类型 | 存储 | 理由 |
|---|---|---|
| 用户/项目/元数据 | PostgreSQL | 关系型数据，需要事务 |
| 话题树 JSON / 对话历史 | PostgreSQL JSONB 字段 | 结构化查询 + 灵活 Schema |
| 会话状态（State Machine） | Redis | 快速读写，支持 TTL 自动过期 |
| 导出文件（JSON/CSV/MD） | 对象存储（S3/R2） | 低成本静态文件存储 |

### 5.3 对话恢复机制

- 对话历史实时存入 `ConversationHistory` 表
- 状态机当前状态存入 Redis，Key 格式：`session:{project_id}:state`
- Redis TTL 设为 24 小时；过期后从数据库恢复
- 用户中途关闭页面后重新访问，系统从数据库加载对话历史和上次状态，继续未完成的流程

## 9. 错误处理与降级策略

| 异常场景 | 处理策略 | 用户感知 |
|---|---|---|
| LLM 输出非法 JSON | 自动重试（最多 3 次），仍失败则降级为 Markdown 格式展示 | "正在优化输出格式，请稍候..." |
| 搜索 API 超时/限流 | 使用缓存数据（若 24h 内有同类查询），否则告知用户并允许手动重试 | "搜索数据获取较慢，可先查看缓存结果" |
| 搜索 API 返回空数据 | LLM 基于自身知识生成初步话题树，标注"未经搜索数据验证"（灰色标记） | "该领域搜索数据较少，以下话题基于 AI 分析，建议人工验证" |
| 用户产品极度小众 | 扩大搜索范围到上层行业/品类，同时提示用户 | "精确搜索量较低，已扩展到上层行业词分析" |
| 对话历史丢失 | 从数据库恢复；若数据库也无记录，引导用户重新开始 | "检测到上次会话异常中断，正在恢复..." |
| 前端渲染大型话题树卡顿 | 超过 100 个节点时分层懒加载，默认折叠二级以下 | 平滑的展开/折叠动画 |

## 14. System Prompt 规范

以下为提供给 LLM 的 System Prompt 核心要素。完整版本需在开发中持续迭代，纳入版本管理。

### 13.1 角色设定

```
你是一位融合了 HubSpot "话题集群模型" 与 Koray Tuğberk GÜBÜR "语义权威地图" 方法论的
顶尖内容营销策略专家。你的任务是通过对话引导用户挖掘产品信息，结合外部搜索数据，
为他们生成高转化、高流量的话题树。
```

### 13.2 交互规则（强约束）

```
<rules>
1. 绝不能在第一轮对话中一次性要求用户提供所有信息。必须采用递进式追问，每轮只问 1-2 个问题。
2. 如果用户回答过于简短，必须追问细节。
   示例：用户说"我做CRM的" → 追问"市面CRM很多，您最想解决销售团队的哪个具体痛点？"
3. 收集到 5+ 维度信息后，必须先向用户发送上下文总结并请求确认，才能进入 S2。
4. 调用搜索 API 前，必须告知用户即将进行数据检索。
5. 生成话题树时，必须为每个树叶标注 search_intent（四分类）和 funnel_stage（TOFU/MOFU/BOFU）。
6. 每棵话题树的 BOFU 内容占比不得低于 15%，确保有转化导向的内容。
7. 输出 JSON 必须严格遵循 generate_topic_tree_json 的 Schema，不得省略任何字段。
</rules>
```

### 13.3 执行流程

```
Step 1 - 破冰：产品一句话介绍 + 产品形态
Step 2 - 用户画像：ICP 是谁？什么行业/职级？
Step 3 - 痛点挖掘：用户最大的痛点是什么？
Step 4 - USP 提炼：同类产品那么多，为什么选你？
Step 5 - 竞品分析：用户现在用什么替代方案？
Step 6 - 市场定位：面向哪些国家/地区？内容用什么语言？
Step 7 - 已有资产：是否有博客/网站？已有多少内容？
Step 8 - 上下文总结与确认
Step 9 - 调用 analyze_market_search_intent，展示数据摘要
Step 10 - 调用 generate_topic_tree_json，输出话题树
Step 11 - 自动生成内链矩阵
Step 12 - 进入迭代模式，响应用户的剪枝/扩写/简报请求
```

## 附录 A：术语表

| 术语 | 定义 |
|---|---|
| Topic Tree（话题树） | 以产品为种子、按 Pillar-Cluster 模型组织的层级内容策略结构 |
| Pillar Page（树干/支柱页） | 覆盖某一大范围话题的"终极指南"型长内容（3000+ 字） |
| Cluster Content（树枝/集群内容） | 围绕某根树干展开的细分子话题集合 |
| Leaf Content（树叶/微意图内容） | 针对单一长尾关键词的具体文章 |
| Topical Authority（话题权威度） | 搜索引擎衡量一个网站在特定领域专业度的指标 |
| TOFU / MOFU / BOFU | 营销漏斗三阶段：认知 / 考虑 / 决策 |
| KD（Keyword Difficulty） | 关键词竞争难度评分，通常 0-100 |
| LSI Keywords | 潜在语义索引关键词，与主题语义相关的词汇 |
| Orphan Page（孤岛页面） | 没有任何其他页面通过内链指向的页面 |
| Keyword Cannibalization（关键词蚕食） | 同一网站多个页面争抢同一关键词排名的问题 |
