# Kingsway Global Voice 页面 PSI 优化建议报告

**页面**: https://cn.kingswayvideo.com/feature/global-voice/  
**检测时间**: 2026-03-03  
**数据来源**: Google PageSpeed Insights (mobile + desktop)

## 二、Core Web Vitals

| 指标 | 移动端 | 桌面端 | 目标/评级 |
|------|--------|--------|-----------|
| **LCP**（最大内容绘制） | **12.4 s** | 2.6 s | <2.5s 为 good，移动端严重超标 |

**结论**：移动端 LCP 12.4 秒是当前最严重问题，会直接拉低转化与跳出率，需优先治理。

## 三、按优先级排序的优化建议

### 1. 严重 (critical) — 优先做

#### 1.1 优化 LCP（最大内容绘制），尤其是移动端

- **问题**：移动端 LCP 12.4 秒，用户感知“首屏很慢”。
- **影响**：LCP 每增加 1 秒，转化率明显下降；>4s 时多数用户会离开。
- **建议**：
  - 为 LCP 元素（多为首屏大图/视频/主标题区域）做**优先加载**：使用 `fetchpriority="high"`、预加载关键图片/字体。
  - 首屏若含视频：改为**懒加载**或**封面图 + 点击播放**，避免自动加载大视频拖慢 LCP。
  - 检查**服务器响应时间 (TTFB)**：若后端或 CDN 慢，先优化接口与缓存。
  - 压缩/优化首屏图片（WebP、合适尺寸），并配合 `loading="eager"` 仅对 LCP 资源使用。
- **预期**：移动端 LCP 可望降至 3–5 秒内，性能分数明显提升。

#### 1.2 减少主线程工作与 JavaScript 执行时间

- **问题**：主线程工作 4.6s（移动）/ 2.9s（桌面）；JS 执行约 1.8s；未使用 JavaScript 约 673 KiB。
- **影响**：页面卡顿、TBT 高、交互响应慢。
- **建议**：
  - **代码分割与懒加载**：非首屏脚本用动态 import 或懒加载，减少首包体积。
  - **移除或延后非关键 JS**：第三方脚本（统计、客服、弹窗等）延后或异步加载。
  - **Tree-shaking 与打包优化**：确保打包工具移除未引用代码，未使用 JS 约 673 KiB 尽量删减。
- **预期**：TBT 与 LCP 会同步改善，性能分数可提升 10–15 分。

---

### 2. 建议 (warning) — 尽快做

#### 2.1 减少未使用的 CSS（约 97 KiB）

- **问题**：存在大量未使用 CSS，增加下载与解析成本。
- **建议**：
  - 使用 PurgeCSS、Critical CSS 或构建阶段只输出当前页面用到的样式。
  - 将首屏关键样式内联或单独小文件优先加载，其余异步加载。
- **预期**：减少首屏体积与渲染阻塞，有助于 FCP/LCP。

#### 2.2 无障碍：链接与按钮需有可识别名称

- **问题**：部分链接无有效文本（`link-name`）；部分 button/link 缺少可访问名称（`aria-command-name`）。
- **影响**：读屏用户无法理解链接用途，SEO 与无障碍均受影响。
- **建议**：
  - 为所有 `<a>` 补全可见或 `aria-label` 文本，避免仅图标无说明。
  - 为图标按钮添加 `aria-label` 或 `title`，确保 `button`/`link` 有可访问名称。
- **预期**：无障碍分数提升，并符合 WCAG 基本要求。

#### 2.3 文档需有主地标 (main landmark)

- **问题**：`Document does not have a main landmark`。
- **建议**：在主体内容外包一层 `<main>`，或为当前主容器加 `role="main"`。
- **预期**：辅助技术可快速定位主内容，无障碍分数提升。

#### 2.4 颜色对比度不足

- **问题**：部分前景/背景对比度不达标（`color-contrast`）。
- **建议**：用 DevTools 或 contrast 工具找出不合格组合，将文字或背景调整至至少 4.5:1（正文）、3:1（大号文字）。
- **预期**：无障碍与最佳实践分数提升。

#### 2.5 使用高效的缓存策略

- **问题**：可节省约 406 KiB 的静态资源未设置长期缓存。
- **建议**：对 JS/CSS/图片等静态资源设置较长 `Cache-Control`（如 1 年），并配合文件名/路径带 hash 做版本管理。
- **预期**：回访用户加载更快，性能与体验更好。

### 3. 可选 (info) — 有余力再做

#### 3.1 控制台错误与废弃 API

- **问题**：存在控制台报错；使用了 1–2 个废弃 API。
- **建议**：在本地与生产环境检查 Console，修复报错；将废弃 API 替换为推荐写法。
- **预期**：最佳实践分数提升，减少未来兼容性风险。

#### 3.2 为大型首方 JavaScript 提供 Source Map

- **问题**：大型首方 JS 缺少 source map。
- **建议**：构建时生成 source map，生产环境可仅对内部/错误上报开放，便于排查问题。
- **预期**：便于调试与监控，不影响终端用户性能。

#### 3.3 桌面端：链接可抓取性与可聚焦的 Skip Link

- **问题**：部分链接不可抓取（`crawlable-anchors`）；Skip link 不可聚焦（`skip-link`）。
- **建议**：确保重要链接为可抓取的 `<a href="...">`；Skip link 使用 `tabindex="0"` 或原生可聚焦元素，并保证键盘可聚焦。
- **预期**：SEO 与键盘/读屏体验更好。

#### 3.4 减少渲染阻塞请求（约 90 ms）

- **问题**：存在可优化的渲染阻塞请求。
- **建议**：将非关键 CSS 改为异步加载（如 `media="print"` + onload 或 link rel preload）；关键 CSS 内联或优先加载。
- **预期**：FCP 略有改善。

## 四、实施顺序建议

1. **第 1 周**：LCP 优化（首屏资源、视频策略、TTFB/CDN）+ 主地标与链接/按钮可访问名称。  
2. **第 2 周**：主线程与 JS 优化（拆包、懒加载、移除未使用 JS）、未使用 CSS 清理、缓存策略。  
3. **第 3 周**：对比度、控制台错误、废弃 API、Skip link 与可抓取链接等收尾项。
